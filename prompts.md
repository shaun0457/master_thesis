# `prompts.py`

## 檔案目的

`prompts.py` 是一個儲存**硬編碼（hardcoded）提示文本**的模組。它包含了為專案中不同代理（DE, DS, ME）在不同模式下（如 `baseline` vs `augmented`）定義的系統提示（System Prompts）字串常數。

在專案的演進過程中，`prompt_builder.py` 和 `prompt/` 資料夾的引入提供了一種更靈活、基於檔案的提示管理方式。因此，`prompts.py` 可以被視為一種**較早期**或**用於快速原型開發/兜底**的提示來源。

## 主要內容

此檔案主要定義了以下幾組 Python 字串常數：

1.  **`DE_SYSTEM_PREAMBLE`**: 為數據工程師（DE）代理提供的一個通用的前導指令，描述了其基本職責、資料庫上下文和必須遵守的核心工作流程與規則。

2.  **`DE_BASELINE_PROMPT`**: DE 代理在「基準模式」（baseline mode）下的提示。這個提示比較簡單，指示 DE 遵循一個線性的工作流程：列出資料表 -> 檢查結構 -> 查詢數據。

3.  **`DE_AUGMENTED_PROMPT`**: DE 代理在「增強模式」（augmented mode）下的提示。這個提示更為嚴格和詳細，強調了「先驗證再查詢」的精確性，並明確指出其最終目標是「交付一個可用的數據集」。

4.  **`DS_BASELINE_PROMPT`**: 為數據科學家（DS）代理在「基準模式」下定義的提示。它設定了較為嚴格的限制，例如：
    *   只允許使用 `pandas` 和 `matplotlib`，禁止使用機器學習套件。
    *   要求第一個行動必須是載入數據。
    *   要求最終的產出必須是一個單一的 JSON 物件。
    *   提供了檔案儲存的命名規範和一個結案 JSON 的範例結構。

5.  **`DS_AUGMENTED_PROMPT`**: DS 代理在「增強模式」下的提示，專為更複雜的機器學習任務設計。它：
    *   賦予 DS 使用 `scikit-learn` 的能力。
    *   提供了一個更專業的工作流程建議（預處理 -> 建模 -> 驗證）。
    *   定義了一個嚴格的**最終交付協議（Final Delivery Protocol）**，要求代理必須將最終的 JSON 報告作為一個獨立的、不帶工具調用的 `AIMessage` 來提交，而不是在 Python 程式碼中 `print` 出來。

6.  **`ME_BASELINE_SYS`** 和 **`ME_AGENTIC_SYS`**: 為機器專家（ME）代理定義的兩種模式下的系統提示，強調了嚴格基於證據回答、在找不到答案時必須承認未知，並為 `agentic` 模式定義了一個詳細的「規劃-搜索-評估-綜合」工作流程。

## 在專案中的角色

在早期或簡化的實驗設置中，各個代理的工作流模組（如 `de_workflow.py`）會直接從 `prompts.py` 導入這些字串，並將它們格式化後作為 System Prompt 傳遞給 LLM。這是一種快速定義代理行為的方式。

隨著 `prompt_builder.py` 的引入，系統的設計趨向於從外部 `.md` 檔案載入提示。在這種情況下，`prompts.py` 中的內容可能扮演著以下角色：

*   **兜底/預設值**: 當 `prompt_builder` 無法在 `prompt/` 資料夾中找到對應的角色卡時，工作流模組可能會回退到使用 `prompts.py` 中定義的這些預設提示。
*   **舊版相容**: 為了與舊的實驗程式碼保持相容。
*   **快速測試**: 在開發新功能時，直接在 Python 檔案中修改字串提示比修改外部檔案更快捷。

總之，`prompts.py` 是代理人「內置指令」的一個靜態儲存庫，與 `prompt_builder.py` 所代表的動態、基於檔案的配置系統形成對比。
