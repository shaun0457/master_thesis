# MAS 系統重構計畫 (MAS Refactoring Plan)

這是一份動態文件，旨在記錄對 MAS 實驗框架的重構思路與計畫。我們可以在討論中隨時更新此文件。

**核心目標：** 提升系統的**模組化**、**可測試性**、**清晰性**和**長期可維護性**。

---

### **任務 0：修復黑板事件日誌記錄 (最高優先級)**

*   **問題**: 使用者回報，儘管程式碼中存在日誌記錄邏輯，但在實際運行中，黑板的讀寫事件（`bb_read`, `bb_write`）並未被成功記錄，導致所有依賴這些事件的指標（如 `reuse_rate`, `t_first_read`）無法計算。
*   **診斷計畫**:
    1.  **注入探針**: 在 `run_logger.py` 的 `emit_bb_read` 和 `emit_bb_write` 函式中加入 `print` 語句作為探針。
    2.  **運行測試**: 執行一次 MAS 互動，觀察主控台是否出現探針的輸出。
    3.  **定位問題**: 
        *   如果**沒有**輸出，說明帶有日誌功能的讀寫函式根本沒有被調用，問題可能出在代理調用了錯誤的或舊版的工具。
        *   如果**有**輸出，說明函式被調用了，但事件未能成功寫入檔案，問題可能在 `emit_event` 的檔案 I/O 邏輯中。
*   **解決方案**: 根據診斷結果，修復對應模組，確保黑板的每一次讀寫都被 `events.jsonl` 忠實記錄。這是所有後續指標分析的基礎，必須最優先解決。

---

### 1. 核心思想：從「隱性上下文」到「顯性依賴注入」

*   **目前狀況**: 系統的多個部分（如 `run_logger`, `bb_tools`）依賴 `os.environ.get("RUN_ID")` 這類的全域環境變數來獲取運行時上下文。
*   **問題**: 這種隱性依賴使得單元測試變得困難，因為需要模擬環境變數，且模組複用性差。
*   **重構方向**:
    1.  **創建一個 `RunContext` 物件**: 在 `chat_cli.py` 啟動時，創建一個 Pydantic 或 dataclass 物件，包含所有運行時配置，例如 `RunContext(run_id=..., db_url=..., policy=...)`。
    2.  **顯式傳遞**: 將這個 `context` 物件作為參數，明確地傳遞給所有需要它的函式（例如 `delegate_to_de(task, state, context)`），徹底移除對 `os.environ` 的依賴。

---

### 2. 架構重組：將「代理」物件化 (Object-Oriented Agents)

*   **目前狀況**: 每個代理的能力（`*_tools.py`）和工作流（`*_workflow.py`）是分離的檔案，由 `router.py` 在運行時動態地 `__import__` 並組合。
*   **問題**: 動態導入功能強大但難以追蹤和維護，IDE 的靜態分析和自動完成功能會失效。
*   **重構方向**:
    1.  **定義 `Agent` 基礎類別**: 創建一個所有專家代理都繼承的抽象基礎類別。
    2.  **實現具體的代理類**: 將每個專家重構為一個類別，例如 `class DataEngineerAgent(Agent):`。
    3.  **依賴注入**: 在代理的建構函式 (`__init__`) 中，將它需要的所有依賴（它的工具集、LLM 執行器、黑板實例等）作為參數傳入。
    4.  **統一的入口**: 每個代理類別都實現一個 `run(task: str) -> AgentOutput` 方法。
    5.  **簡化 Router**: `router.py` 不再需要動態導入，只需從一個代理註冊表（字典）中獲取代理實例，並調用其 `.run()` 方法。

---

### 3. 工作流簡化：抽象出通用的 ReAct 循環

*   **目前狀況**: `me_workflow.py`, `de_workflow.py`, `ds_workflow_s2.py` 各自實現了一套 LangGraph 的 ReAct 循環，存在程式碼重複。
*   **問題**: 修改通用的 ReAct 邏輯（如錯誤處理）時，需要在多個地方同步修改。
*   **重構方向**:
    1.  **創建 `build_generic_react_graph()` 函式**: 建立一個只負責構建通用 ReAct 循環（如 `think` -> `act` -> `think`）的函式。
    2.  **在代理類中使用**: 每個代理的 `.run()` 方法內部調用這個通用圖，並傳入**自己獨有的 `agent_executor`**（即綁定了該代理特定工具和提示的 LLM）。
    3.  **結果**: 將工作流的「結構」與代理的「具體能力」分離，用一個可複用的圖結構取代多個重複的工作流檔案。

---

### 4. 強化黑板：使其成為真正的「單一」事實來源

*   **目前狀況**: 黑板的狀態同時存在於記憶體中的 `state['blackboard']` 和檔案系統中的 `registry.json`，`bb_snapshot_text` 需要複雜的邏輯來合併兩者。
*   **問題**: 雙重狀態源增加了系統的複雜性和出錯的可能性。
*   **重構方向**:
    1.  **將 `Blackboard` 物件化**: 將 `bb_tools.py` 重構為一個 `Blackboard` 類別，在系統啟動時只實例化一次。
    2.  **移除記憶體鏡射**: 廢除在 `state['blackboard']` 中保存副本的做法。
    3.  **統一讀寫介面**: 任何模組對黑板的讀寫操作，都必須通過注入的 `blackboard` 實例的方法（如 `blackboard.read_facts()`）來進行，這些方法直接操作底層的 `registry.json` 檔案。
    4.  **結果**: 邏輯變得極其清晰，`registry.json` 成為了唯一的、真正的 Single Source of Truth，大大提升了系統的穩定性和可預測性。

---

### 5. 配置管理：將「實驗變數」外部化

*   **目前狀況**: 實驗的關鍵變數（如 `POLICY`）被硬編碼在 `prompt_builder.py` 等多個檔案中，修改不便。
*   **問題**: 難以自動化地運行多組使用不同配置的對比實驗。
*   **重構方向**:
    1.  **`chat_cli.py` 作為唯一入口**: 利用 `argparse` 將所有實驗變數（如 `--policy`, `--model`, `--temperature`）都定義為命令列參數。
    2.  **寫入 `RunContext`**: 在 `chat_cli.py` 啟動時，將從命令列解析到的所有參數值都存入我們在第一點中提議的 `RunContext` 物件。
    3.  **動態傳遞**: 將 `RunContext` 物件作為參數一路傳遞下去。任何需要讀取配置的函式（如 `get_system_prompt`）都從這個 `context` 物件中讀取，而不是從檔案內的常數或環境變數中讀取。
    4.  **結果**: 實現了配置與程式碼的完全分離，運行任何實驗組合都只需改變命令列參數，無需修改程式碼。

### 6. 新增指標建議 (待日誌問題修復後實現)

#### 6.1 更高階的知識流動指標

*   **知識傳遞鏈長度 (Knowledge Cascade Length)**
    *   **定義**: 一個關鍵資訊從被創建到最終被使用的過程中，平均經過了多少次代理的「交接」。
    *   **測量**: 在實驗後，分析 `events.jsonl`，追蹤 `artifact_id` 在 `bb_write` 和 `bb_read` 事件之間的傳遞路徑長度。

*   **知識中介中心性 (Knowledge Brokerage Centrality)**
    *   **定義**: 哪些代理在知識流動網絡中扮演了關鍵的「橋樑」角色。
    *   **測量**: 將代理視為節點，每一次 `bb_read` 視為一條從「作者」到「讀者」的邊，建立知識流動圖。計算圖中每個節點的「介數中心性 (Betweenness Centrality)」。

#### 6.2 更深層的穩定性指標

*   **策略遵循度 / 協議違規率 (Protocol Adherence)**
    *   **定義**: 代理的行為在多大程度上遵循了當前 `policy` 所規定的行為模式。
    *   **測量**: 在 MAS 運行過程中，於 `router.py` 等關鍵節點加入檢查邏輯，判斷代理行為是否「違規」（例如，在 `PTOW` 模式下發生 P2P 委派），若違規則記錄一條特定的 `compliance` 事件。

*   **任務焦點漂移指數 (Topic Drift Index)**
    *   **定義**: 團隊的討論焦點是否隨時間推移而偏離了最初的用戶問題。
    *   **測量**: 在 MAS 運行過程中，於 `supervisor_node` 中，每次 LLM 產生決策時，計算其「意圖」的語意嵌入向量與「初始問題」向量的餘弦相似度，並記錄該相似度分數的變化趨勢。
