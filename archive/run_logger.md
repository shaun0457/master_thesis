# `run_logger.py`

## 檔案目的

`run_logger.py` 提供了一套全面的、結構化的**事件日誌記錄系統**。它的核心目的是捕獲 MAS（多代理系統）在一次運行（run）中發生的所有關鍵行為，並將它們以**每行一個 JSON 物件**的格式（JSON Lines, `.jsonl`）寫入到日誌檔案中。這為後續的數據分析、行為追蹤、性能評估和錯誤診斷提供了原始的、機器可讀的數據來源。

此模組是整個系統**可觀測性（Observability）**的基石。

## 主要功能與組件

1.  **`RunLogger` 類 (Class)**:
    *   這是一個較早期的、基於類別的日誌記錄器實現。它在實例化時與一個特定的 `run_id` 綁定。
    *   提供了多個方法（如 `new_task`, `close_task`, `agent_node`, `tool_exec`）來記錄不同類型的事件。
    *   它將所有事件聚合在記憶體中的 `self.data` 字典裡，並在程式退出時通過 `atexit.register(self.flush)` 將整個 `data` 物件一次性寫入到一個大的 JSON 檔案中。
    *   在目前的專案結構中，它的部分功能（如事件記錄）已被更輕量級的獨立函式所取代，但它仍然可能在某些舊的或特定的實驗流程中被使用。

2.  **獨立的事件發射函式 (Standalone Emitter Functions)**:
    *   這是目前更主流的日誌記錄方式，提供了一系列更專注、更解耦的函式。
    *   **`emit_event(e: dict, ...)`**: 這是所有事件記錄的**最底層核心函式**。它接收一個事件字典 `e`，為其自動添加 `run_id`, `event_id`, `timestamp` 等元數據，然後將其作為一行附加到 `events.jsonl` 檔案中。
    *   **`log_message(...)`**: 專門用於記錄代理的發言事件。
    *   **`note_tool_call(...)`**: 專門用於記錄一次工具調用的摘要資訊（名稱、成功狀態、延時等）。
    *   **`emit_bb_write(...)`** 和 **`emit_bb_read(...)`**: 專門用於記錄對黑板的讀寫操作，包含了 `topic_id`, `artifact_id` 等關鍵的追蹤資訊。
    *   **`emit_compliance(...)`**: 用於記錄與「操守」或「協定」相關的事件，例如，是否發生了 P2P 的私下通訊（side-channel）。
    *   **`emit_outcome(...)`**: 在運行結束時，記錄最終的結果，如成功分數、總回合數等。

3.  **運行元數據管理 (`begin_run`, `end_run`, `emit_run_meta`)**:
    *   這些函式負責管理與一次運行本身相關的元數據（存放在 `run_meta.json` 中），而不是單個事件。
    *   `begin_run`: 記錄運行的開始時間和一些初始配置（如 `task_id`, `model_name` 等）。
    *   `end_run`: 記錄運行的結束時間。

## 日誌檔案結構

此模組的產出主要有兩類檔案，都位於 `RUNS_DIR`（預設為 `/mnt/data/runs`）下的對應 `run_id` 目錄中：

*   **`events.jsonl`**: 核心的事件日誌。每一行都是一個 JSON 物件，代表一個獨立的事件，如一次工具調用、一次黑板寫入、一次委派等。這種格式非常適合用 `pandas.read_json(..., lines=True)` 或其他大數據工具進行流式處理和分析。
*   **`run_meta.json`**: 儲存這次運行的總體元數據，如使用的模型、提示詞版本、開始/結束時間等。
*   **`outcome.json`**: 儲存這次運行的最終結論性指標，如成功分數。

## 設計理念

*   **事件溯源（Event Sourcing）**: 系統的設計思想接近於事件溯源。它不直接修改一個中央的狀態資料庫，而是將每一次狀態的「變更意圖」或「行為」記錄為一個不可變的事件。系統的最終狀態可以通過重播（replay）所有事件來重建。
*   **去耦合**: 獨立的 `emit_*` 函式使得專案中的任何模組都可以輕鬆地記錄日誌，而無需知道 `RunLogger` 類別的具體實現。它們只需要調用相應的函式即可，這大大降低了耦合度。
*   **數據的原始性與豐富性**: 日誌系統旨在捕獲盡可能原始和豐富的資訊。例如，`note_tool_call` 不僅記錄成功與否，還記錄了參數的摘要和延時；`emit_bb_write` 不僅記錄寫入，還記錄了寫入的 `artifact_id` 和預期的擁有者 `intended_owner`。這為後續進行深入、多維度的分析提供了可能。

## 依賴關係

*   **Python 函式庫**: `os`, `json`, `time`, `hashlib`, `uuid`, `traceback`, `contextlib`。
*   **內部模組**:
    *   `common.py`: 依賴其 `ensure_run_id` 函式來確保在分散的調用中也能獲得正確的 `run_id`。

## 在專案中的角色

`run_logger.py` 是整個 MAS 實驗框架的「**飛行數據記錄儀（黑盒子）**」。它忠實地記錄下系統運行時發生的一切，是保證實驗**可重複性、可追溯性和可分析性**的基礎設施。如果沒有這個模組，所有的代理行為和協作過程都將是瞬時的、無法觀測的，從而無法進行任何科學的評估和改進。
